# Multi-Sided Market Exchange PBC

## Stable Identity

The `multi_sided_market` PBC is a commerce mesh packaged business capability for multi-party exchange. It supports marketplaces where participants trade, barter, sell, book, rent, and loan goods or services. The pbc key is stable and the package owns the code under `src/pyAppGen/pbcs/multi_sided_market`. The capability is deliberately separate from catalog, inventory, payment, tax, fraud, and identity packages. Those dependencies are represented through APIs, projections, or AppGen-X events rather than shared table access.

## Owned Boundary

The owned table boundary is explicit. The PBC owns participant_profile, marketplace_listing, listing_asset, service_offer, availability_window, booking_reservation, rental_contract, loan_agreement, barter_offer, trade_order, sale_order, exchange_proposal, escrow_account, settlement_instruction, dispute_case, reputation_signal, market_rule, market_parameter, schema_extension, governed_model. Runtime event infrastructure adds `multi_sided_market_appgen_outbox_event`, `multi_sided_market_appgen_inbox_event`, and `multi_sided_market_appgen_dead_letter_event`. Foreign or shared tables are not used. Product, inventory, payment, tax, fraud, and access-policy facts arrive through declared consumed events or through named APIs. This keeps participant, listing, exchange, escrow, settlement, reputation, and dispute state isolated and independently scalable.

## Schema Generation

Schema generation creates a migration and model for every owned table. The migration creates prefixed tables only, and the model manifest proves every model table begins with the `multi_sided_market_` prefix. Relationships are represented inside the package boundary through listing, exchange, escrow, settlement, booking, rental, loan, barter, trade, sale, dispute, and reputation identifiers. The schema extension hook allows governed JSON-style or typed extension fields on owned tables without changing external PBC storage.

The owned schema is intentionally not a generic key/value catalog. Participant profiles store role sets, verification level, trust score, default currency, and policy context. Marketplace listings store title, listing kind, supported exchange modes, price, currency, and publication timing. Listing assets store quantity, condition, location, and attributes. Service offers store provider, service type, duration, and service area. Availability windows store start/end timing, capacity, and reserved counts. Bookings, rentals, and loans carry their own lifecycle fields including hold expiry, condition records, collateral, due dates, and risk scores. Barter, trade, sale, exchange proposal, escrow, settlement, dispute, and reputation tables each contain their own operational fields so generated apps can persist real market workflows without borrowing another PBC table.

Rules, parameters, schema extensions, and governed models are first-class owned records. Market rules store policy bodies, compiled hashes, effective dates, and rule type. Parameters store bounded values and change ownership. Schema extensions store table name, field definitions, version, and compatibility posture. Governed models store purpose, version, metrics, and governance status for matching, trust, pricing, dispute triage, or document-understanding models. This gives generated applications a place to understand and implement market configuration instead of hard-coding policy in route handlers.

Commands include participant verification, listing publication, listing asset registration, service offer creation, availability-window publication, trade order placement, barter matching, exchange proposal preparation, sale completion, booking reservation, rental start, loan issue, escrow preparation, escrow release policy compilation, escrow release, settlement preparation, reputation signal recording, dispute opening, dispute resolution, and market-clearing projection publication. The workbench and analytical queries are read-only and emit no outbox event. Each route has an operation contract, idempotency key convention, permission, table scope, event contract, and owned datastore plus outbox transaction boundary.

Command methods validate the market boundary by mutating only owned tables and writing AppGen-X outbox records. Read routes are explicitly query operations and do not publish events. The generated route contract includes a stable idempotency convention in the form `multi_sided_market:<operation>:{request_id}` so API adapters can deduplicate commands for participant onboarding, listing publication, booking, rental, loan, barter, trade, sale, escrow, settlement, and dispute workflows. The service manifest also publishes the external dependency shape: catalog, inventory, payment, tax, fraud, and access-policy information is consumed through declared APIs or events only.

The service layer covers table-stakes market workflows: participant onboarding, role management, seller and buyer flows, provider service offers, borrower and lender flows, asset registration, inventory-aware listing, availability management, booking reservation, rental contracting, goods loaning, barter negotiation, trade ordering, exchange proposal review, sale checkout handoff, escrow hold/release, settlement instruction, reputation scoring, dispute resolution, market-clearing projection, policy registration, parameter updates, schema extension registration, and workbench querying.

## Events And Handlers

The PBC emits typed events: MarketParticipantVerified, MarketListingPublished, ListingAssetRegistered, ServiceOfferCreated, AvailabilityWindowPublished, TradeOrderPlaced, BarterOfferMatched, ExchangeProposalPrepared, SaleCompleted, BookingReserved, RentalStarted, LoanIssued, EscrowOpened, EscrowReleasePolicyCompiled, EscrowReleased, MarketSettlementPrepared, ReputationSignalRecorded, MarketDisputeOpened, MarketDisputeResolved, MarketClearingProjected. It consumes ProductPublished, InventoryPoolChanged, PaymentCaptured, TaxCalculated, FraudRiskScored, AccessPolicyChanged. Eventing uses AppGen-X with an outbox, inbox, idempotency store, retry policy, and dead-letter table. Handlers are idempotent: duplicate idempotency keys return duplicate evidence instead of replaying a mutation. Unknown consumed events go to the dead-letter plan with retry evidence. The runtime does not expose stream-engine picker choices to users.

Event payloads are scoped to market facts owned by this PBC: participant verification, listing publication, asset registration, availability, exchange proposal, order placement, barter matching, sale completion, booking reservation, rental start, loan issue, escrow policy, escrow release, settlement preparation, reputation signal, dispute opening, dispute resolution, and clearing projection. Consumed events are treated as projections or policy inputs rather than table-sharing permission. Product publication can enrich listing eligibility, inventory pool changes can affect availability, captured payments can satisfy escrow release prerequisites, tax calculations can annotate settlement plans, fraud scores can influence trust gates, and access policy changes can refresh command permissions. Handler plans expose retry count, dead-letter table, idempotency key, consumed event type, and side-effect-free validation evidence.

## UI Workbench And Permissions

The UI workbench includes MarketExchangeWorkbench, ListingConsole, BookingRentalCalendar, EscrowSettlementConsole, DisputeResolutionBoard. It surfaces participant onboarding, listing management, exchange negotiation, booking and rental calendars, escrow and settlement controls, dispute triage, event monitoring, and configuration governance. Permissions are multi_sided_market.read, multi_sided_market.create, multi_sided_market.update, multi_sided_market.approve, multi_sided_market.settle, multi_sided_market.admin. RBAC descriptors map create, update, approve, settle, read, and admin actions to explicit permission strings. The assistant surface is part of the UI contract and contributes scoped skills to the composed application assistant.

The workbench view is an operational screen, not a marketing page. It surfaces listing counts, booking volume, active rentals, loan exposure, dispute counts, outbox health, dead-letter health, configuration status, rule status, and matching recommendations. Listing consoles support goods, services, bundles, time slots, rental assets, loanable items, and barter/trade candidates. Booking and rental calendars expose capacity, holds, expiry, return due dates, condition capture, and collateral state. Escrow and settlement consoles show hold reason, release policy hash, payment reference, tax reference, settlement destination, and blocked-release explanations. The dispute board shows evidence, triage classification, counterparty history, policy references, proposed resolutions, and audit trail.

AI assistance is a first-class UI and DSL contribution. The PBC surfaces a chatbot interface with scoped skills for participant onboarding, listing drafting, service offer setup, booking adjustment, rental condition review, loan term explanation, barter equivalence analysis, trade matching, sale checkout guidance, escrow release reasoning, dispute evidence intake, reputation explanation, rule authoring, parameter tuning, schema-extension planning, and governed CRUD. The assistant can accept documents or instructions, extract structured market facts, propose safe datastore mutations, and return a plan that names the command, permission, owned tables, idempotency key, and expected event. In a composed application, these skills merge into the single application assistant under the `multi_sided_market_skills` namespace.

## Rules Parameters Configuration

The PBC can understand and implement rules, parameters, and configuration. Rules include listing policy, exchange policy, escrow release policy, dispute policy, and reputation policy. Parameters include commission rate, escrow hold days, maximum rental days, trust threshold, and loan collateral rate. Configuration requires datastore backend, event topic, retry limit, default currency, escrow hold days, and max rental days. Allowed datastore backends are PostgreSQL, MySQL, and MariaDB, and eventing is AppGen-X.

Rules are compiled into auditable hashes and can be versioned without crossing the PBC boundary. Listing rules govern category eligibility, prohibited goods or services, required verification levels, document requirements, and disclosure fields. Exchange rules govern barter tolerances, trade matching constraints, price bounds, capacity limits, location restrictions, carbon-aware fulfillment preferences, and counterparty trust thresholds. Escrow rules govern hold duration, partial release, dispute lock, payment evidence, tax evidence, and fulfillment confirmation. Reputation rules govern score deltas, decay, evidence weighting, dispute penalties, and privacy-preserving proof references. Parameter contracts define bounded values so generated apps can reject unsafe configuration before a command executes.

The advanced runtime models multi-party exchange graphs, barter equivalence, combinatorial matching, liquidity and trust scoring, availability-aware booking, rental collateral, loan risk, escrow policy compilation, real-time market clearing, counterfactual pricing, semantic instruction parsing, autonomous dispute triage, anomaly detection, privacy-preserving reputation, carbon-aware fulfillment choices, and cross-PBC integration.

Advanced matching treats listings, assets, offers, availability, participants, and exchange modes as a graph. It can recommend two-party or multi-party exchanges, score barter equivalence, produce counteroffers, compare sale versus rental versus loan economics, and explain why a candidate was selected or rejected. Booking optimization considers time windows, capacity, participant trust, service duration, and hold expiry. Rental and loan workflows model condition, collateral, due dates, risk, and return obligations. Escrow compilation turns market policies into release checks and escrow release records prove whether required payment, fulfillment, and dispute-window checks were satisfied. Dispute triage interprets evidence and routes cases to resolution paths while keeping an audit trail. Counterfactual simulation allows a generated app to ask how price, trust, collateral, hold duration, tax, or availability changes would affect a match or settlement. Fraud and collusion detection examines repeated exchange pairs and dispute rate. Privacy-preserving reputation proofs disclose a score band without exposing every exchange. Carbon-aware fulfillment selection chooses the lowest-carbon viable meetup or delivery option.
